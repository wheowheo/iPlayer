# iPlayer 개발 문서

## 렌더링 아키텍처

iPlayer는 두 가지 렌더링 모드를 지원하며, 재생 중에도 실시간 전환이 가능하다.

### 렌더 모드 전환
- **단축키**: `R`
- **우클릭 메뉴**: 렌더 모드 > CVDisplayLink / Thread
- **정보 오버레이** (Tab): `Render:` 항목에 현재 모드 표시
- 기본값: `CVDisplayLink`

---

## CVDisplayLink 렌더링

### 개요

`CVDisplayLink`는 macOS의 Core Video 프레임워크가 제공하는 **디스플레이 vsync 동기화 타이머**다.
디스플레이의 실제 리프레시 레이트(60Hz, 120Hz, ProMotion 등)에 정확히 맞춰 콜백을 호출하므로,
비디오 프레임을 **화면 갱신 시점에 정확히** 전달할 수 있다.

### 동작 원리

```
디스플레이 vsync (60Hz = 16.67ms 간격)
    │
    ▼
CVDisplayLink 콜백 (별도 고우선순위 스레드)
    │
    ├─ 마스터 클럭 계산 (오디오 PTS 또는 벽시계)
    ├─ 프레임 큐에서 적정 프레임 선택
    │   ├─ 너무 늦은 프레임 → 드롭 (droppedFrames++)
    │   ├─ 약간 늦은 프레임 → 즉시 표시 (따라잡기)
    │   ├─ 적정 타이밍 → 표시
    │   └─ 아직 이른 프레임 → 대기 (다음 vsync에서 재시도)
    ├─ FPS 측정
    └─ DispatchQueue.main.async → CALayer.contents 업데이트
```

### API 흐름

```swift
// 1. 생성
CVDisplayLinkCreateWithActiveCGDisplays(&link)

// 2. 콜백 등록
CVDisplayLinkSetOutputCallback(link, callback, context)
// callback 시그니처:
// (CVDisplayLink, UnsafePointer<CVTimeStamp>, UnsafePointer<CVTimeStamp>,
//  CVOptionFlags, UnsafeMutablePointer<CVOptionFlags>, UnsafeMutableRawPointer?) -> CVReturn

// 3. 시작 (vsync마다 콜백 호출 시작)
CVDisplayLinkStart(link)

// 4. 정지
CVDisplayLinkStop(link)
```

### CVDisplayLink 콜백 파라미터

| 파라미터 | 타입 | 설명 |
|---------|------|------|
| `displayLink` | `CVDisplayLink` | 현재 DisplayLink 인스턴스 |
| `inNow` | `UnsafePointer<CVTimeStamp>` | 현재 시각 (호스트 타임) |
| `inOutputTime` | `UnsafePointer<CVTimeStamp>` | 다음 vsync 시각 (프레임이 실제로 화면에 나타나는 시점) |
| `flagsIn` | `CVOptionFlags` | 입력 플래그 |
| `flagsOut` | `UnsafeMutablePointer<CVOptionFlags>` | 출력 플래그 |
| `context` | `UnsafeMutableRawPointer?` | 사용자 컨텍스트 (PlayerController 포인터) |

### 메모리 관리

CVDisplayLink 콜백은 C 함수 포인터이므로 Swift 클로저 캡처를 사용할 수 없다.
대신 `Unmanaged`를 통해 `self`를 컨텍스트로 전달한다:

```swift
// 시작 시: retain (+1)
let ptr = Unmanaged.passRetained(self).toOpaque()
displayLinkSelfPtr = ptr  // 포인터 저장
CVDisplayLinkSetOutputCallback(link, callback, ptr)

// 콜백 내: retain count 변경 없이 접근
let controller = Unmanaged<PlayerController>.fromOpaque(context!).takeUnretainedValue()

// 정지 시: release (-1)
Unmanaged<PlayerController>.fromOpaque(ptr).release()
```

**주의**: `passRetained`으로 얻은 포인터를 반드시 저장해두고, `stopDisplayLink()`에서 정확히 한 번 `release`해야 한다.
정지 시 `passRetained`을 다시 호출하면 새로운 retain이 발생하여 메모리 릭 또는 이중 해제가 된다.

### Thread 모드와의 차이

| 항목 | CVDisplayLink | Thread (기존) |
|------|--------------|---------------|
| **타이밍 소스** | 디스플레이 vsync 하드웨어 | `Thread.sleep` (소프트웨어) |
| **호출 주기** | 정확히 디스플레이 주사율 | sleep 정밀도에 의존 (~1ms 오차) |
| **CPU 사용** | vsync 간격에만 깨어남 | 60% sleep + 40% spin 대기 |
| **티어링** | 없음 (vsync 동기) | 가능 (비동기 렌더) |
| **프레임 드롭** | 최소화 | sleep 오차로 인한 불필요한 드롭 가능 |
| **스레드** | Core Video가 관리하는 고우선순위 스레드 | 직접 생성한 Thread |
| **배터리** | 효율적 (idle 시 완전 대기) | 비효율적 (spin 대기) |

### CVDisplayLink를 선택한 이유

1. **vsync 정합**: macOS의 WindowServer는 vsync에 맞춰 화면을 합성한다. Thread.sleep으로는 이 타이밍을 정확히 맞출 수 없어 프레임이 한 vsync 늦게 표시되거나, 같은 프레임이 두 번 표시될 수 있다.

2. **4K 대응**: 4K 60fps = 16.67ms 간격. Thread.sleep의 ~1ms 오차가 누적되면 주기적으로 프레임 드롭이 발생한다. CVDisplayLink는 하드웨어 타이밍이므로 이 문제가 없다.

3. **가변 주사율(ProMotion)**: MacBook Pro의 ProMotion(120Hz)이나 외장 모니터의 다양한 주사율에 자동 적응한다. Thread 모드는 수동으로 주사율을 감지해야 한다.

4. **에너지 효율**: CVDisplayLink는 프레임을 그릴 필요가 없을 때(일시정지 등) 콜백에서 즉시 리턴하면 CPU를 소비하지 않는다.

### 한계 및 주의사항

- **macOS 전용**: iOS/tvOS에서는 `CADisplayLink`를 사용해야 한다 (유사하지만 API가 다름)
- **Deprecated 경고**: macOS 15+에서 `CVDisplayLinkCreateWithActiveCGDisplays`가 deprecated 되었다. 향후 `CADisplayLink` (macOS 14+)로 마이그레이션 가능하다.
- **멀티 모니터**: 창이 다른 모니터로 이동하면 해당 디스플레이의 CVDisplayLink로 전환해야 최적이다. 현재는 활성 디스플레이 기준으로 생성한다.
- **콜백 스레드**: CVDisplayLink 콜백은 메인 스레드가 아닌 Core Video 전용 스레드에서 호출된다. UI 업데이트는 반드시 `DispatchQueue.main.async`로 전달해야 한다.

## 프레임 렌더링 파이프라인

```
[Read Thread]                    [Render (CVDisplayLink/Thread)]         [Main Thread]
     │                                      │                                │
     ├─ Demuxer.readPacket()                │                                │
     ├─ VideoDecoder.decode()               │                                │
     ├─ FrameRingBuffer.append() ──────────►│                                │
     │                                      ├─ computeMasterClock()          │
     │                                      ├─ selectFrame()                 │
     │                                      ├─ DispatchQueue.main.async ────►│
     │                                      │                                ├─ CALayer.contents = frame
     │                                      │                                ├─ onTimeUpdate
     │                                      │                                └─ onSubtitleUpdate
```

### 프레임 선택 알고리즘

```swift
for each frame in queue:
    diff = frame.pts - masterClock

    if diff < -3 * frameDuration:
        // 심하게 늦음 → 드롭
        drop(frame)

    else if diff < -1 * frameDuration:
        // 약간 늦음 → 즉시 표시 (따라잡기)
        show(frame)

    else if diff <= 0.5 * frameDuration:
        // 적정 타이밍 → 표시
        show(frame)

    else:
        // 아직 이름 → 대기
        break
```

### A/V 동기화

- **마스터 클럭**: 오디오가 있으면 `AudioOutput.compensatedPTS` (오디오 큐 레이턴시 보상 포함)
- **폴백**: 오디오 없는 영상은 `CACurrentMediaTime()` 기반 벽시계 클럭
- **드리프트 측정**: `avSyncDrift = frame.pts - audioOutput.compensatedPTS`
- 정보 오버레이에서 `A/V Drift` 값으로 실시간 모니터링 가능 (±ms 단위)

## 향후 개선 가능 사항

### CADisplayLink 마이그레이션 (macOS 14+)
```swift
// CVDisplayLink 대체
let displayLink = CADisplayLink(target: self, selector: #selector(tick))
displayLink.add(to: .main, forMode: .common)
// 장점: 메인 런루프 통합, Unmanaged 불필요, iOS와 API 통일
// 단점: 메인 스레드 실행이므로 무거운 작업 시 UI 블로킹 가능
```

### Metal 렌더링
현재 `CALayer.contents`에 직접 `CVPixelBuffer`/`CGImage`를 할당하는 방식이다.
Metal을 사용하면 GPU에서 직접 텍스처를 렌더링할 수 있어 CPU 부하를 줄일 수 있다:
- `CVMetalTextureCache`로 CVPixelBuffer → MTLTexture 변환 (제로카피)
- `CAMetalLayer`로 vsync 동기화된 Metal 렌더링
- 색공간 변환, 자막 합성 등을 GPU 셰이더로 처리 가능

## macOS 그래픽 렌더링 아키텍처

### Windows Surface ↔ macOS 대응

| Windows | macOS | 역할 |
|---------|-------|------|
| **DXGI Surface** | **IOSurface** | GPU 메모리에 할당된 픽셀 버퍼. 프로세스 간 공유 가능 |
| **Direct3D Texture** | **MTLTexture** | GPU 텍스처 (렌더 타겟/샘플링) |
| **DirectComposition Visual** | **CALayer** | 합성(compositing) 단위. 트리 구조 |
| **DWM (Desktop Window Manager)** | **WindowServer** | 최종 화면 합성 데몬 |
| **DXGI SwapChain** | **CAMetalLayer** | 더블/트리플 버퍼링 + vsync 제출 |
| **GDI DC / GDI+ Graphics** | **CGContext** | 2D 소프트웨어 래스터라이저 |
| **Direct2D** | **Core Graphics (Quartz 2D)** | 2D 벡터 렌더링 (GPU 가속) |
| **WIC (Windows Imaging)** | **CGImage / CIImage** | 이미지 디코딩/처리 |
| **DirectShow / Media Foundation Surface** | **CVPixelBuffer** | 비디오 프레임 컨테이너 |

### 렌더링 파이프라인 전체 구조

```
┌─────────────────────────────────────────────────┐
│                    Display                       │
│              (물리 화면 출력)                      │
└────────────────────┬────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────┐
│              WindowServer                        │
│   macOS 시스템 합성기 (DWM에 해당)                  │
│   - 모든 윈도우의 CALayer 트리를 수집                │
│   - GPU에서 최종 합성 (Metal 기반, macOS 10.14+)    │
│   - vsync에 맞춰 디스플레이에 스캔아웃               │
│   - 투명도, 그림자, Mission Control 등 처리         │
└────────────────────┬────────────────────────────┘
                     │ IOSurface (프로세스 간 공유)
┌────────────────────▼────────────────────────────┐
│           Core Animation (QuartzCore)            │
│   - CALayer 트리를 GPU 커맨드로 변환                │
│   - 레이어 속성: contents, transform, opacity 등   │
│   - 암시적/명시적 애니메이션                         │
│   - 백킹 스토어: IOSurface                        │
└──┬──────────┬──────────┬───────────┬────────────┘
   │          │          │           │
   ▼          ▼          ▼           ▼
 CALayer  CAMetalLayer  CAOpenGL   CALayer
 .contents  (Metal)    GLLayer    (CG 렌더)
   │          │          │           │
   ▼          ▼          ▼           ▼
CGImage/   MTLTexture  OpenGL    CGContext
CVPixelBuf  SwapChain   FBO      → IOSurface
```

### IOSurface — macOS의 "Surface"

Windows DXGI Surface에 가장 가까운 것. **GPU 메모리에 할당된 픽셀 버퍼**로, 핵심 특징:

- **프로세스 간 공유**: 앱 → WindowServer 간 제로카피 전달 (Mach port로 공유)
- **GPU/CPU 양쪽 접근**: Metal 텍스처로 매핑하거나 CPU에서 직접 읽기/쓰기
- **CALayer의 백킹 스토어**: 모든 CALayer는 내부적으로 IOSurface에 렌더링됨

```
App Process                    WindowServer Process
┌──────────┐                  ┌──────────────┐
│ CALayer   │── IOSurface ───►│  합성 엔진     │──► Display
│ .contents │   (GPU 메모리)   │  (Metal)      │
└──────────┘                  └──────────────┘
     ↑ 같은 GPU 메모리를 양쪽 프로세스가 참조 (제로카피)
```

### CVPixelBuffer — 비디오 프레임 Surface

비디오 디코딩의 출력 단위. iPlayer에서 VideoToolbox가 반환하는 것:

```
VideoToolbox HW 디코더
    │
    ▼
CVPixelBuffer (내부 = IOSurface 백킹)
    │
    ├─ CALayer.contents = pixelBuffer  ← 현재 iPlayer 방식 (제로카피)
    │     Core Animation이 IOSurface를 직접 WindowServer에 전달
    │
    ├─ CVMetalTextureCache → MTLTexture  ← Metal 방식 (제로카피)
    │     GPU에서 직접 셰이더로 처리 후 CAMetalLayer에 표시
    │
    └─ CVPixelBufferLockBaseAddress → CPU 접근  ← SW 처리 (카피 발생)
          swscale 등으로 포맷 변환
```

**핵심**: HW 디코딩된 CVPixelBuffer는 이미 IOSurface 백킹이므로, `CALayer.contents`에 넣으면 **GPU 메모리 복사 없이** 바로 화면에 나간다.

### CALayer — 합성 단위 (Visual Tree)

Windows의 DirectComposition Visual과 같은 역할:

```swift
// 각 CALayer = 하나의 합성 Surface
window.contentView.layer          // 루트
  ├── videoLayer                  // 비디오 (IOSurface 직접 참조)
  ├── controlBar.layer            // 컨트롤 UI
  │     ├── playButton.layer
  │     ├── seekBar.layer
  │     └── volumeSlider.layer
  ├── subtitleLabel.layer         // 자막
  └── infoOverlay.layer           // 정보
```

WindowServer가 이 트리를 받아서 GPU로 한 번에 합성한다. 앱은 개별 레이어만 업데이트하면 됨.

### 렌더링 경로 비교

| 경로 | 파이프라인 | GPU 카피 | 지연 |
|------|-----------|---------|------|
| **HW디코딩 → CALayer** | VT → CVPixelBuffer(IOSurface) → CALayer → WindowServer | **0회** | 최소 |
| **HW디코딩 → Metal** | VT → CVPixelBuffer → MTLTexture → CAMetalLayer → WindowServer | **0회** (텍스처 매핑) | 최소 + 셰이더 |
| **SW디코딩 → CGImage → CALayer** | FFmpeg → RAM버퍼 → CGImage → CALayer(IOSurface 업로드) | **1회** | 중간 |
| **SW디코딩 → Metal** | FFmpeg → RAM버퍼 → MTLTexture 업로드 → CAMetalLayer | **1회** | 중간 |

### WindowServer 합성 과정

```
매 vsync (16.67ms @60Hz):

1. 각 앱의 CALayer 트리에서 변경된 IOSurface 수집
2. Metal 컴퓨트/렌더 패스로 합성:
   - 윈도우 순서대로 블렌딩
   - 투명도, 그림자, 모서리 라운딩
   - Mission Control / Spaces 변환
3. 최종 프레임버퍼를 디스플레이 컨트롤러에 스캔아웃
```

### iPlayer 현재 렌더링 경로

```
[VideoToolbox]                          [swscale]
HW 디코더                                SW 디코더
     │                                      │
     ▼                                      ▼
CVPixelBuffer                            CGImage
(IOSurface 백킹)                      (RAM → GPU 업로드)
     │                                      │
     └──────────┬───────────────────────────┘
                │
                ▼
     CALayer.contents = frame
                │
                ▼  (Core Animation이 IOSurface를 WindowServer에 제출)
         WindowServer 합성
                │
                ▼
           Display 출력
```

HW 디코딩 경로에서는 **GPU 메모리 카피가 0번** — VideoToolbox가 디코딩한 IOSurface를 WindowServer가 바로 합성한다. 이것이 M1에서 4K가 가벼운 이유다.
